## Домашнее задание 3: API, DTO, мапперы и persist в `localStorage`

### Цель
Перевести конвертер с локальных моков на данные из API, сохранить пользовательское состояние между перезагрузками и отделить серверный контракт от клиентских моделей через DTO и мапперы.

Эта домашка опирается на темы лекции:
- правильное разделение состояния: что хранить в `state`, а что вычислять;
- `useState` vs `useReducer`;
- controlled vs uncontrolled компоненты;
- обзор `Context API`;
- базовое использование `useEffect` для получения данных.

### Стек
- `Vite + React + TypeScript`
- `SCSS Modules`
- `fetch`, `axios` или `ky` для HTTP-запросов
- `Vitest + React Testing Library`

### Источник данных
Данные нужно получать не из моков, а из локального backend-приложения в папке `server`.

### Контракт API
Используйте DTO, приходящие с backend.

`CurrencyDto`:
- `code: string`
- `name: string`
- `description: string`
- `symbol: string`

`PriceChangeDto`:
- `purchasedCurrencyCode: string`
- `paymentCurrencyCode: string`
- `price: number`
- `dateTime: string` (ISO 8601)

### Структура папок
Важно соблюдать понятную структуру проекта. Минимально ожидается разбиение по зонам ответственности:
- `src/api` - функции HTTP-запросов к backend;
- `src/dto` - типы DTO, соответствующие ответам API;
- `src/mappers` - преобразование `DTO -> client model`;
- `src/models` - клиентские модели данных, с которыми работает UI;
- `src/hooks` - хуки, связанные с загрузкой данных и `localStorage`;
- `src/components` - React-компоненты;
- `src/pages` или `src/features` - композиция экранов/фич, если такая структура уже используется в проекте.


### Что нужно реализовать
1. Подключить фронтенд к локальному backend из папки `server`.
2. Получать список валют с API, а не из локальных моков.
3. Получать данные по курсам с API, а не из локальных моков.
4. Завести отдельные TypeScript-типы для DTO:
   - `CurrencyDto`;
   - `PriceChangeDto`.
5. Завести отдельные клиентские модели данных для UI.
6. Реализовать мапперы:
   - `CurrencyDto -> Currency`;
   - `PriceChangeDto -> PriceChange`.
7. В компонентах и бизнес-логике использовать только клиентские модели, а не DTO напрямую.
8. Сохранять в `localStorage` состояние, важное для пользователя:
   - выбранную валюту `from`;
   - выбранную валюту `to`;
   - введенную сумму;
   - состояние блока `More about ...`, если оно у вас сохраняется локально.
9. При старте приложения восстанавливать это состояние из `localStorage`.
10. Реализовать безопасную обработку невалидных данных в `localStorage`:
   - если JSON поврежден, приложение не падает;
   - если структура данных не совпадает с ожидаемой, используется fallback;
   - если в `localStorage` сохранены валюты, которых нет в новом ответе API, приложение корректно выбирает валидную пару.
11. Получение данных из API реализовать через отдельный слой доступа к данным, а не напрямую внутри JSX.
12. Использовать `useEffect` для загрузки данных осознанно и минимально необходимым образом.

### Как работать с backend из папки `server`
Backend написан на ASP.NET Core и лежит в папке `server`.

Как запустить:
1. Убедиться, что установлен .NET SDK.
2. Открыть терминал в папке `server`.
3. Выполнить команду `dotnet run`.
4. После запуска backend будет доступен по адресу `http://localhost:5081`, а Swagger - по адресу `http://localhost:5081/swagger`.

Основные эндпоинты:
- `GET /Currency` - получить список валют;
- `GET /Currency/{code}` - получить одну валюту по коду;
- `GET /prices?paymentCurrency=XXX&purchasedCurrency=YYY&fromDateTime=...&toDateTime=...` - получить историю курсов для пары валют.

Как использовать в клиенте:
1. При инициализации приложения загрузить список валют через `GET /Currency`.
2. Для выбранной пары валют запрашивать курсы через `GET /prices`.
3. Для расчета конвертации использовать актуальную запись из полученной истории курсов.
4. Если запрос завершился ошибкой, показать пользователю безопасный fallback: сообщение об ошибке, состояние загрузки или пустой результат без падения приложения.

Примечания по backend:
- сервер периодически генерирует новые значения курсов;
- эндпоинт `/prices` возвращает историю изменений, поэтому на клиенте нужно выбрать актуальную запись;
- для разработки можно смотреть формат ответов через Swagger или файл `server.http`.

### Тесты
Минимальный набор тестов:
- unit-тесты на мапперы `CurrencyDto -> Currency` и `PriceChangeDto -> PriceChange`;
- тест на восстановление состояния из `localStorage` при старте;
- тест на сохранение состояния после изменения пары валют или суммы;
- тест на безопасную обработку битого JSON в `localStorage`;
- тест на поведение при невалидной структуре данных в `localStorage`;
- тест на корректную работу UI после получения данных с API или мокнутого HTTP-ответа.

### Ограничения
- Не использовать глобальный state-менеджер только ради этой задачи.
- Не использовать библиотеки автогенерации мапперов.
- Не использовать `localStorage` как замену API-кэшу для всех серверных данных.
- Не хранить в `localStorage` полный список валют и всю историю курсов, если они могут быть заново получены с сервера.
- Не смешивать DTO и клиентские модели в одном типе.
- Важно соблюдать структуру папок и разделение ответственности между слоями.

### Критерии приемки
- Приложение получает данные из backend в папке `server`.
- Запросы к API вынесены в отдельный слой.
- DTO типизированы отдельно от клиентских моделей.
- Данные из API проходят через явные мапперы до попадания в UI.
- После перезагрузки страницы выбранная пара валют и сумма восстанавливаются из `localStorage`.
- Приложение не падает при битых или невалидных данных в `localStorage`.
- Структура проекта остается понятной и разделенной по ответственности.
- Тесты проходят через `npm run test`.
